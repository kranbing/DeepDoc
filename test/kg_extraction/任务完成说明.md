# 成员一：知识抽取与清洗模块 - 任务完成说明

## 模块概述

本模块负责从文档 chunk 中抽取实体和关系，是知识图谱构建流水线的第一道工序。采用"规则抽取 + LLM 抽取"双通道策略，在保证召回率的同时控制 API 调用成本。

---

## 完成的任务

### 1. 实体抽取功能 (`entity_extractor.py`)

#### 规则抽取（10 种模式）

| 编号 | 模式 | 正则/逻辑 | 置信度 | 示例 |
|------|------|-----------|--------|------|
| 1 | 括号缩写 | `([A-Z][a-zA-Z\s]{2,30})\s*\(([A-Z]{2,6})\)` | 0.9 | "Long Short Term Memory (LSTM)" |
| 2 | 大写缩写 | `\b[A-Z]{2,6}\b` | 0.7 | BERT, RAG, NMT |
| 3 | 大写短语 | `\b[A-Z][a-z]+(?:\s+[A-Z][a-z]+)+\b` | 0.5 | Neural Machine Translation |
| 4 | 引号术语 | `"([^"]{2,50})"` | 0.6 | "attention mechanism" |
| 5 | 已知数据集 | 40+ 个 NLP 数据集名称库（大小写不敏感匹配） | 0.85 | SQuAD, MS MARCO, ImageNet |
| 6 | 已知指标 | 20+ 个评估指标名称库 | 0.85 | F1, BLEU, ROUGE-L |
| 7 | 章节标题 | chunk 的 sectionTitle 字段 | 1.0 | "摘要"、"相关工作" |
| 8 | 中文技术术语 | 预定义的 40+ 个中文 NLP/AI 术语 | 0.8 | 深度学习、注意力机制、知识图谱 |
| 9 | 中文引号术语 | `[""「」]([^""「」]{2,30})[""「」]` | 0.6 | "检索增强生成" |
| 10 | 中英对照 | `([一-鿿]{2,15})\s*[（(]\s*([A-Za-z][\w\-]*)\s*[）)]` | 0.85 | 知识图谱(Knowledge Graph) |

#### LLM 抽取（DeepSeek API）

- **模型**：deepseek-chat，temperature=0.1，max_tokens=1024
- **输出格式**：强制 JSON（`response_format: json_object`）
- **Prompt 设计**：system prompt 指定 8 种实体类型（Method/Dataset/Metric/Concept/Task/Tool/Author/Organization），要求保留原文语言（中文保持中文）
- **批量策略**：每个文档每 3 个 chunk 调用一次 LLM，最多 8 次调用，平衡成本与覆盖
- **容错**：API 超时 60s，失败时返回空列表不中断流程

#### 实体元数据

每个实体附带完整溯源信息：
- `source_doc_id`：来源文档 ID
- `source_chunk_id`：来源 chunk ID
- `source_chunk_ids`：去重后合并的所有来源 chunk ID 列表
- `source_section`：来源章节标题
- `source`：抽取来源标记（`rule_paren_acronym` / `llm` 等）
- `confidence`：置信度评分（0.0-1.0）

### 2. 关系抽取功能 (`relation_extractor.py`)

#### 模式匹配（10 种正则模式）

| 关系类型 | 触发关键词 | 示例 |
|---------|-----------|------|
| USES | use/utilize/employ/leverage | "BERT uses attention" |
| EVALUATES_ON | evaluate/test/benchmark/assess | "evaluated on SQuAD" |
| ACHIEVES | achieve/attain/reach/score | "achieves 92.3% F1" |
| BASED_ON | based on/build on/extend/improve | "based on Transformer" |
| PROPOSES | propose/introduce/present/design | "proposes a new method" |
| BELONGS_TO | belong to/type of/kind of | "is a type of RNN" |
| DEPENDS_ON | depend on/require/need | "requires pre-training" |
| COMPARES_WITH | compare/outperform/surpass | "outperforms BERT" |
| CONTAINS | contain/include/comprise | "contains 12 layers" |
| PART_OF | is part of/belongs to | "is part of the pipeline" |

- **模糊匹配**：实体名称支持子串匹配，处理 "BERT" 匹配 "BERT-base" 等变体

#### 共现分析

基于同一 chunk 内实体共现推断隐含关系：
- Method + Dataset → `EVALUATES_ON`（置信度 0.5）
- Method + Metric → `ACHIEVES`（置信度 0.4）
- Method + Method → `COMPARES_WITH`（置信度 0.35）

#### 结构关系抽取

从文档 LAD 结构中提取层级关系：
- Document `CONTAINS` Section（置信度 1.0）
- Section `CONTAINS` Chunk（置信度 1.0）
- 支持多级 section 嵌套（通过 parentId 构建层级树）

#### LLM 关系分类

- 对模式匹配和共现分析未覆盖的实体对，调用 DeepSeek API 进行关系分类
- 每个 chunk 最多 5 对实体调用 LLM
- 返回结构化 JSON：`{relation, confidence, evidence}`
- 置信度 < 0.3 的结果自动过滤

### 3. 清洗模块 (`cleaner.py`)

#### 实体清洗流水线

```
原始实体 (855) → 噪声过滤 → 名称规范化 → 精确去重 → 模糊合并 → 清洗后实体 (231)
```

**噪声过滤规则**：
- 空白/纯标点实体
- 长度 ≤ 1 的实体；长度 ≤ 2 且非全大写
- 纯数字（如 "2023"）
- 英文停用词（180+ 个）
- 超长实体（> 100 字符，疑似抽取错误）
- HTML 片段（包含 `<` `>` 标签，如 `<td>...</td>`）

**去重策略**：
- 第一轮：精确去重 — 规范化名称（去空白、去首尾标点）+ 类型相同 → 保留高置信度
- 第二轮：模糊合并 — 同类型实体名称相似度 ≥ 0.85 → 合并为一个（保留最长名称 + 最高置信度）
- 缩写识别：单个大写单词与多词短语的首字母缩写匹配（如 "NMT" ↔ "Neural Machine Translation"，相似度 0.95）

**ID 重分配**：去重后按 `ent_{docId}_{序号:04d}` 格式重新分配 ID

#### 关系清洗

- 过滤置信度 < 0.25 的关系
- 去除自环（source_id == target_id）
- 去除空 source_name / target_name
- 验证关系类型是否在 10 种合法类型中
- 验证 source_id 和 target_id 在有效实体集合中（允许 section_/chunk_/doc_ 前缀的结构节点）
- 去重：同 (source, target, relation) 保留高置信度

#### 清洗报告

自动生成结构化报告，包含：
- 实体/关系的原始数量、清洗后数量、去除数量、去重率
- 实体类型分布（清洗前后对比）
- 抽取来源分布（rule_* vs llm）
- 关系类型分布

### 4. 抽取演示 (`extract_demo.py`)

- 可对单个文档运行完整的实体+关系抽取流程
- 输出 `demo_entities.json`、`demo_relations.json`、`demo_extraction_report.json`

## 技术亮点

1. **双通道抽取**：规则抽取零成本、高精度；LLM 抽取高召回、语义理解强。两者互补，覆盖率远超单一方案。
2. **中文适配**：针对中文技术文档设计了中文技术术语、中文引号、中英对照等 3 种专属规则模式。
3. **渐进式 LLM 调用**：按 chunk 采样调用（每 3 个 chunk 调 1 次），用最少的 API 调用获取最大覆盖面。
4. **智能去重**：支持缩写识别（首字母匹配）和模糊合并，有效处理同一实体的多种表述形式。
5. **完整溯源**：每个实体保留 source_chunk_id / source_section / source 等字段，可追溯到原始文本位置。
