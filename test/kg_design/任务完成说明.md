# 成员三：图谱设计、流程与展示模块 - 任务完成说明

## 模块概述

本模块负责知识图谱的整体架构设计、构建流水线编排和交互式可视化展示。作为项目的"总设计师"，定义了图谱的 Schema、Pipeline 和前端展示方案，并实现了完整的端到端可视化系统。

---

## 完成的任务

### 1. 知识图谱结构设计 (`kg_schema.md`)

#### 实体类型（10 种）

| 类型 | 说明 | 属性 | 示例 |
|------|------|------|------|
| Document | 文档根节点 | docName, abstract | "DeepDoc-文档理解与..." |
| Section | 文章节标题 | parentId, level, sectionId | "摘要"、"相关工作" |
| Chunk | 文本块 | sectionId, content | chunk_p0003_c002 |
| Method | 方法/模型 | description, confidence | 检索增强生成（RAG） |
| Dataset | 数据集 | description | SQuAD, MS MARCO |
| Metric | 评估指标 | description | F1, BLEU, ROUGE-L |
| Concept | 概念/术语 | description | 知识图谱、深度学习 |
| Task | 任务 | description | 问答、机器翻译 |
| Tool | 工具/框架 | description | PyTorch, HuggingFace |
| Author | 作者 | description | 樊亦简、于智通 |
| Organization | 机构 | description | 计算机与人工智能学院 |

#### 关系类型（10 种）

| 关系 | 语义 | 触发条件 | 示例 |
|------|------|---------|------|
| CONTAINS | 包含 | 文档结构层级 | Document → Section → Chunk |
| PROPOSES | 提出 | 方法创新 | RAG PROPOSES 双层索引 |
| USES | 使用 | 技术依赖 | BERT USES Attention |
| EVALUATES_ON | 评估于 | 基准测试 | BERT EVALUATES_ON SQuAD |
| ACHIEVES | 达到 | 性能指标 | BERT ACHIEVES F1=92.3 |
| BASED_ON | 基于 | 改进关系 | RoBERTa BASED_ON BERT |
| BELONGS_TO | 属于 | 分类关系 | LSTM BELONGS_TO RNN |
| DEPENDS_ON | 依赖 | 前置条件 | 微调 DEPENDS_ON 预训练 |
| COMPARES_WITH | 对比 | 性能对比 | BERT COMPARES_WITH GPT |
| PART_OF | 属于 | 组成关系 | Encoder PART_OF Transformer |

#### 节点属性设计

通用属性：`id`, `type`, `name`, `description`, `source_doc_id`, `source_chunk_id`, `confidence`

溯源属性：`source`（抽取来源）, `source_chunk_ids`（合并后来源列表）, `source_section`（来源章节）

结构属性：`parentId`（Section 父节点）, `level`（Section 层级）, `sectionId`（Chunk 所属章节）

#### 边属性设计

`source_id`, `target_id`, `relation`, `confidence`, `evidence`（证据文本）, `source_chunk_id`（来源 chunk）

### 2. 构建流程设计 (`kg_pipeline.md`)

设计了完整的 7 步构建流水线：

```
Step 1: 文档读取     → 从 LAD chunk.json 加载文档（chunks, sections, abstract）
Step 2: 实体抽取     → 规则抽取 + LLM 抽取（成员一实现）
Step 3: 关系抽取     → 模式匹配 + 共现分析 + 结构关系 + LLM（成员一实现）
Step 4: 清洗         → 噪声过滤 + 去重 + 模糊合并 + 关系验证（成员一实现）
Step 5: 图谱写入     → 写入 JsonGraphStore，添加 Document/Section/Chunk 结构节点（成员二实现）
Step 6: 查询测试     → 5 类查询验证图谱质量（成员二实现）
Step 7: 输出展示     → 生成 JSON、HTML、Markdown 三种格式（本模块实现）
```

#### 配置参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `max_docs` | 5 | 最大处理文档数 |
| `max_llm_calls` | 8 | 每文档最大 LLM 调用数 |
| `llm_batch_size` | 3 | 每 N 个 chunk 调用一次 LLM |
| `MIN_CONFIDENCE` | 0.25 | 关系最低置信度阈值 |
| `DEDUP_THRESHOLD` | 0.85 | 实体名称相似度合并阈值 |

### 3. 测试案例集 (`test_cases.json`)

8 个测试案例，覆盖从单元测试到集成测试的完整链路：

| 编号 | 测试类型 | 测试内容 | 验证点 |
|------|---------|---------|--------|
| TC-01 | 单元测试 | 单 chunk 实体抽取 | 规则 + LLM 抽取结果结构正确 |
| TC-02 | 单元测试 | 共现实体关系抽取 | Method-Dataset/Metric 正确推断 |
| TC-03 | 单元测试 | 实体去重 | 同名同类型合并，保留高置信度 |
| TC-04 | 单元测试 | 噪声实体过滤 | 空实体、停用词、HTML 片段被过滤 |
| TC-05 | 单元测试 | JSON 存储读写 | save → load 往返一致 |
| TC-06 | 单元测试 | 一跳查询 | 返回正确的邻居列表 |
| TC-07 | 单元测试 | 两实体路径查询 | BFS 找到正确路径 |
| TC-08 | 集成测试 | 完整 pipeline | 端到端运行，输出有效图谱 |

### 4. Pipeline 入口 (`run_kg_pipeline.py`)

作为整个知识图谱系统的统一入口脚本，编排成员一（抽取清洗）和成员二（存储查询）的完整流程。

#### 核心流程

1. **文档加载**（`load_documents`）：从 LAD 数据目录读取 `lad_chunk.json`，提取 chunks、docName、abstract、sections
2. **文档处理**（`process_document`）：对每篇文档执行实体抽取 → 关系抽取 → 清洗 → 添加结构节点（Document/Section/Chunk）
3. **图谱构建**（`build_graph`）：将所有文档结果写入 JsonGraphStore
4. **查询测试**（`run_query_tests`）：执行 5 类查询验证
5. **报告生成**（`generate_extraction_reports`）：生成成员一的抽取统计报告
6. **可视化输出**：调用 `generate_html_preview` 和 `generate_markdown_preview`
7. **前端部署**：将图谱 JSON 复制到前端可访问路径

#### 关键实现细节

- **实体 ID 重映射**：清洗后实体 ID 变化，需要将关系的 source_id/target_id 通过实体名称映射到新 ID
- **结构节点注入**：为每个 Document/Section/Chunk 创建对应的图节点，附带 parentId/level 等层级信息
- **chunk 名称清理**：去除 HTML 标签，截取前 80 字符作为显示名
- **多路径输出**：同时输出到 `test/kg_output/` 和 `test/lad_test/`（前端访问路径）

### 5. 可视化后端生成 (`visualize_graph.py`)

Python 脚本，从 `kg_graph.json` 生成两种静态预览文件。

#### HTML 预览（`graph_preview.html`）

- **技术栈**：D3.js v7 力导向图，纯 SVG 渲染
- **布局算法**：`d3.forceSimulation` + `forceLink` + `forceManyBody` + `forceCenter` + `forceCollision`
- **交互功能**：
  - 节点拖拽（`d3.drag`）
  - 缩放平移（`d3.zoom`，scaleExtent 0.1-8）
  - 悬停 tooltip（显示名称、类型、描述）
  - 搜索高亮（输入框实时过滤，非匹配节点降低透明度）
- **视觉编码**：
  - 节点颜色：10 种类型各一种颜色
  - 节点大小：Document 12px, Section 8px, 其他 6px
  - 边颜色：按关系类型着色（`d3.schemeTableau10`）
- **侧边栏**：统计信息 + 搜索框 + 类型图例 + 关系类型列表
- **自包含**：单个 HTML 文件，仅依赖 D3.js CDN

#### Markdown 预览（`graph_preview.md`）

- 表格形式的图谱摘要：节点类型分布、关系类型分布
- 前 20 个实体和前 20 条关系的示例表格

### 6. 前端交互式可视化页面 (`kg_graph_frontend.html`)

完全重写的多视图知识图谱可视化系统，纯 Canvas 实现（无外部依赖，兼容中国网络环境）。

#### 三视图架构

**视图一：文档结构（旭日图 Sunburst）**

- **数据源**：Document → Section → Chunk 层级结构
- **布局**：Canvas 绘制同心圆环
  - 第一环：Document（中心圆点）
  - 第二环：按 parent 分组的 Section（扇区，颜色按角度分配）
  - 第三环：Chunk（细分扇区，灰色薄环）
- **交互**：悬停高亮路径，点击显示详情
- **搜索**：输入关键词高亮匹配章节
- **侧边栏**：可折叠的章节树，显示每节 chunk 数量

**视图二：实体关系（力导向图 + 渐进展开）**

- **初始状态**：展示 12 个有直接关系的核心实体（7 个 Method 完全子图 + 2 个 Author + 1 个 Metric）
- **渐进展开**：点击任意节点 → 展开其关联实体
  - 通过直接实体间边查找邻居
  - 通过 `source_chunk_ids`（共享文本块）查找同章节实体
  - 通过 `source_section`（共享章节）查找同章节实体
  - 自动添加来源章节节点作为上下文
- **收起功能**：再次点击非核心节点可收起展开的实体
- **节点标识**：
  - ★ 金色：核心节点（初始连接实体，不可收起）
  - ◉ 蓝色 +：可展开（有未显示的关联实体）
  - − 灰色：可收起
- **物理模拟**：
  - 中心引力 + 碰撞检测 + 弹簧力
  - 类型聚类（同类型节点相互吸引）
  - 阻尼系数 0.82，模拟自然冷却
- **边渲染**：贝塞尔曲线 + 关系标签 + 按关系类型着色
- **BELONGS_TO 合成边**：实体与所属章节之间的紫色虚线

**视图三：实体总览（Dashboard）**

- **左侧**：按类型分组的实体列表（可搜索、按关联数排序）
- **右侧**：选中实体的详情面板（类型、关联数、描述、所在章节、关联节点列表）
- **底部**：实体类型关系热力图（Canvas 绘制矩阵，颜色深度表示关系数量）

#### 通用功能

- **顶栏**：三个 Tab 切换 + 搜索框（Ctrl+K 聚焦）+ 统计指标
- **侧边栏**：类型图例 + 类型开关（可隐藏/显示特定类型）
- **Canvas 渲染**：`devicePixelRatio` 适配高 DPI 屏幕
- **缩放控制**：+/−/适配视图 三个按钮
- **Tooltip**：悬停显示节点名称、类型、关联数、描述
- **详情面板**：点击节点弹出浮层，显示完整信息和关联列表

### 7. 前端集成

在 DeepDoc 现有前端中集成知识图谱入口：

- **`web/index.html`**：在 OCR/CHUNK/LAD 按钮旁添加 KG 按钮
- **`web/app.js`**：KG 按钮点击事件 → 创建 iframe 加载 `/api/kg/graph.html`
- **`backend/main.py`**：
  - `GET /api/kg/graph.json`：返回 `test/lad_test/kg_graph.json` 文件
  - `GET /api/kg/graph.html`：返回 `test/kg_design/kg_graph_frontend.html` 文件

## 技术亮点

1. **三视图分离**：将"文档树 + 稀疏实体关系"这一棘手的可视化问题拆分为三个独立视图，每个视图解决不同展示需求，避免了传统力导向图在 96% 边为 CONTAINS 时的混乱。
2. **渐进展开**：实体关系视图采用"核心实体 → 点击展开"的渐进式探索模式，从 12 个核心实体出发，通过共享 chunk/section 逐步发现更多关联实体，解决了 116 个孤立实体的展示问题。
3. **纯 Canvas 实现**：不依赖 D3.js 或其他外部库，完全兼容中国网络环境（无需翻墙下载 CDN）。物理模拟、碰撞检测、贝塞尔曲线绘制均手工实现。
4. **物理模拟优化**：中心引力 + 碰撞 + 弹簧 + 类型聚类四重力场，阻尼冷却，支持拖拽固定节点。针对 700+ 节点优化了 O(n²) 碰撞检测的性能。
5. **端到端自动化**：`run_kg_pipeline.py` 一个脚本完成从文档读取到可视化的全部流程，包含 API key 检测、进度输出、错误处理。
6. **前端无缝集成**：通过 iframe 嵌入现有 DeepDoc 前端，与 OCR/CHUNK/LAD 视图使用相同的交互模式。
