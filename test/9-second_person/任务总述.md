# 第二人任务总述

## 1. 任务原文

第二人：实现任务类型分发机制，对接不同 Prompt 模板，chunk 缓存设计，对接大模型的异常处理，针对不同任务类型设计不同模型参数（温度、max_tokens 等），记录不同参数模式下的结果影响，生成测试报告。

本项目中进一步细化为：主系统需要自动对用户问题分类，之后根据任务类型决定对应的提示词和 RAG 或 LADRAG。右上角 RAG 切换增加 AUTO，默认 AUTO；AUTO 时自动处理，不是 AUTO 时强制使用用户选择的检索模式。同时要求真实对接大模型，并提供参数测试脚本。

## 2. 任务原文中做出来的展示内容有哪些

- 实现了任务类型自动分发。
- 实现了 AUTO / RAG / LAD 三种检索模式。
- AUTO 模式下自动判断任务类型，并选择 RAG 或 LADRAG。
- 非 AUTO 模式下强制使用用户选择的检索模式，但仍按任务类型分发 Prompt 和模型参数。
- 对接不同任务类型的 Prompt 模板和输出 Schema。
- 按任务类型配置不同模型参数，包括 `temperature`、`top_p`、`max_tokens`、`timeout_seconds`、`max_retries`。
- 对 DeepSeek 真实调用增加异常处理，包括缺少 API key、timeout、429/5xx 重试、空响应和 JSON 解析失败。
- 前端右上角检索模式按钮改为三态：`AUTO -> RAG -> LAD -> AUTO`。
- 编写真实大模型参数测试脚本，并已经运行生成测试报告。

## 3. 任务的实现文件位置与功能

- `backend/services/task_dispatcher.py`
  - 主系统任务分发模块。
  - 根据用户问题分类为 `fact_qa`、`local_evidence_qa`、`multi_evidence_qa`、`method_qa`、`comparison_qa`、`dataset_qa`。
  - 输出完整 `TaskRoute`，包含任务类型、检索模式、系统提示词、任务提示词、响应契约和输出 Schema。

- `backend/services/model_config.py`
  - 模型参数配置模块。
  - 为不同任务类型配置不同模型参数。
  - 为参数测试提供 `stable`、`balanced`、`exploratory` 三种模式。

- `backend/clients/deepseek_client.py`
  - DeepSeek 真实调用客户端。
  - 支持 `model`、`temperature`、`top_p`、`max_tokens`、`timeout_seconds`、`max_retries`。
  - 实现 API 异常、超时、限流、服务端错误、空响应和 JSON 解析错误处理。

- `backend/services/qa_service.py`
  - 将任务分发出来的 Prompt、Schema 和模型参数接入真实问答调用。
  - 保留模型原始结构化输出。
  - 返回任务路由、模型参数和后续第三人需要的证据链结构。

- `backend/main.py`
  - 主接口薄接入。
  - `ask-selection` 接口支持 `retrievalMode=auto|rag|lad`。
  - 保持逻辑模块化，不把任务分发细节堆在 `main.py`。

- `web/index.html`
  - 右上角检索按钮默认改为 `AUTO`。

- `web/app.js`
  - 检索模式三态切换。
  - 向后端发送当前 `retrievalMode`。

- `test/9-second_person/model_configs.json`
  - 参数测试脚本使用的三种参数模式配置。

- `test/9-second_person/run_model_param_test.py`
  - 第二人真实大模型参数测试脚本。
  - 读取 QASPER 测试样本。
  - 对不同参数模式调用 DeepSeek。
  - 记录格式合规、引用字段、延迟、答案长度、错误信息和真实回答。
